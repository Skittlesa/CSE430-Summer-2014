There are two issues here. One is the side effect of a boolean test, the other is the truth table outcome.
While it is true that if readerCount > 0 we always want to V(mutex), because we won't be releasing a thread with dequeue, and it is true that if dequeue()==False we want to V(mutex) it is also true that we don't want the side effect of performing the dequeue unless readerCount==0. Unless the language specification specifically states that the second boolean case in an 'or' statement will not be tested if the first case is true you can't assume that the second case (dequeue) isn't tested each time causing, in this case, an unwanted writer thread to be released when readers are still active.

We already achieved starvation free in part 2. Part 3 is going for fairness.

If you look closely at the output your program generated you'll see that reader 2 starts reading followed by writer 0 writing before reader 2 leaves the critical section. Writer 0 should not have been able to write until all readers were done reading. There is no starvation because there is no exclusive access to the larger critical section. In the real world this would result in data corruption.

For part 3 you are going to use enqueue() in place of both P(rsem) and P(wsem) since you want to block the threads all on the same queue in the order they appear.
You will use dequeueReader() or dequeueWriter() in place of V(rsem) and V(wsem) respectively since dequeue unblocks a thread only if it is the type you asked for.
Your code would be altered to look something like this:

Reader Entry
    P(mutex);
    if (wc>0) or (wwc>0) then
        begin
           wrc++
           V(mutex);
           queue.enqueue();
           wrc--;
        end;
    rc++;
    if (queue.dequeueReader()=0) then
        V(mutex);

Reader Exit
    P(mutex);
    rc--;
    if (rc>0) then
        V(mutex);
    else
        Release();

Writer Entry
    P(mutex);
    if (rc>0)or(wc>0) then
        begin
            wwc++;
            V(mutex);
            queue.enqueue();
            --wwc;
        end;
    wc++;
    V(mutex);

Writer Exit
    P(mutex);
    wc-- ; 
    Release();

Release
    if (wrc>0 or wwc>0)
        while (true)
            begin
                if (queue.dequeueWriter()=0) then
                    if (queue.dequeueReader()=0) then
                        continue;
                break;
            end
    else
        V(mutex);

You may recall that in class I discussed the fact there are multiple sources of race conditions. Even though the updating of counters is protected by critical sections so that they remain correct, the reporting of the reader count rc is not, nor is there a serialization of the output of the data to the screen. The example you supplied exhibits both of these cases. Even though the reader reports that it is reading before the writer reports that it is done writing, that is an example of output produced by separate threads that did not appear in the correct order since the output itself is not generated in a critical section. The same with the count. It was not reported from within a critical section. Ask me in class and I'll use your example.
